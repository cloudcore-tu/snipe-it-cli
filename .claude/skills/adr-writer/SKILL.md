# ADR Writer

設計判断を Architecture Decision Record として記録する。
ADR は `docs/adr/` ディレクトリに連番で管理する。

## ワークフロー

### 1. 既存 ADR の番号を確認し、ADR ファイルを作成する

```bash
ls docs/adr/
```

最大の番号を確認し、次の連番を使う。
ファイル名: `NNNN-kebab-case-title.md`（例: `0001-use-cobra-for-cli.md`）

### 2. テンプレートに従って記述する

```markdown
# ADR-NNNN: タイトル

- **ステータス**: 提案 | 承認 | 却下 | 廃止
- **日付**: YYYY-MM-DD

## コンテキスト

この決定が必要になった背景。どんな問題を解決しようとしているか。

## 検討した選択肢

### A. 選択肢名

説明と pros/cons。

### B. 選択肢名

説明と pros/cons。

## 決定

何を選んだか、なぜ選んだか。具体的な理由を書く。

## 結果

この決定の結果、何が変わるか。影響やトレードオフ。

## 未決事項

（なければ省略）
```

## デフォルトステータス

- AI が作成する ADR のデフォルトステータスは **提案** とする

## 記述のガイドライン

- **コンテキストは事実ベースで書く**: 「〜だと思う」ではなく「〜という制約がある」
- **選択肢は公平に評価する**: 採用しなかった選択肢にも正当な pros を書く
- **決定理由は具体的に**: 「総合的に判断して」ではなく、何がどう優れていたか
- **ステータスの更新**: 状況が変わったら廃止にして新しい ADR を書く
